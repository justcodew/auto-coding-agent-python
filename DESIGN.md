## 核心哲学

### AI 时代的程序员定位

本项目的核心假设是：

> **AI 已经能够编写绝大部分代码。如果不行，不是 AI 的问题，是使用者的问题。AI 不是程序员的 bottleneck，程序员错误和老旧的工作方式是 AI 的 bottleneck。**

这意味着：

- **写代码本身的价值正在贬值**。未来程序员的竞争力不在于"写得多快"，而在于"描述得多准"。
- **程序员的工作内容正在向上游迁移**：从"如何实现"转向"实现什么"和"为什么这样实现"。
- **与 AI 协作的能力成为核心技能**：如何拆解需求、如何验收产出、如何在 AI 偏离时纠正方向。

### 设计目标

基于以上认知，本系统的设计目标不是"替代程序员"，而是：

1. **降低沟通成本**：让 AI 之间的协作像人类团队一样结构化
2. **提高可追溯性**：每一步决策都有据可查
3. **控制风险**：防止 AI 在无人监督时失控
4. **保持透明**：所有中间产物对人类可见、可干预

## 为什么选择多智能体架构

### 单一 Agent 的局限性

在原项目的单 Agent 模式中，一个 Agent 需要同时承担：

- 理解需求
- 拆解任务
- 编写代码
- 自我检查
- 修复错误

这种模式的问题：

| 问题 | 表现 |
|:---|:---|
| **认知负载过高** | Agent 在同一上下文中切换多种角色，容易混淆 |
| **自我审查盲区** | 同一个 Agent 很难发现自己的错误 |
| **纠错能力弱** | 陷入错误时缺乏外部视角来打破循环 |
| **Prompt 臃肿** | 需要在一个 Prompt 中塞入所有角色的指令 |

### 多智能体的优势

将不同职责分配给专职 Agent：

```
单 Agent 模式：
用户 → [一个万能的 Agent] → 代码

多 Agent 模式：
用户 → Planner → Coder → Debugger → Reviewer → 代码
        ↑         ↓        ↑         ↓
        └─────────┴────────┴─────────┘
                   （反馈循环，各司其职）
```

| 优势 | 说明 |
|:---|:---|
| **专注** | 每个 Agent 的 Prompt 更短、更聚焦 |
| **制衡** | Debugger 天然与 Coder 对立，形成检查-平衡机制 |
| **可替换** | 可以单独优化某个角色，不影响整体 |
| **可观测** | 每个环节的输入输出都是独立的检查点 |

### 为什么是四个角色

选择 Planner、Coder、Debugger、Reviewer 四个角色，是因为这个分工映射了人类软件团队的经典结构：

| 人类角色 | AI 角色 | 核心职责 |
|:---|:---|:---|
| 产品经理 / 架构师 | Planner | 理解需求，设计实现路径 |
| 开发工程师 | Coder | 将设计转化为代码 |
| 测试工程师 | Debugger | 验证代码正确性 |
| 技术负责人 | Reviewer | 最终质量把关 |

这个分工已被软件工程实践验证有效，移植到 AI 系统中同样适用。

## 为什么选择文件交互模式

### 对比：框架模式 vs 文件模式

| 维度 | 框架模式（如 MS Agent Framework） | 文件交互模式 |
|:---|:---|:---|
| **耦合度** | 高，Agent 依赖框架内存状态 | 低，仅依赖文件约定 |
| **可观测性** | 需框架提供调试工具 | 直接查看文件，天然透明 |
| **可中断性** | 中断后状态难恢复 | 随时可停，状态全在文件中 |
| **学习成本** | 需学习框架 API | 只需理解文件结构 |
| **扩展性** | 受框架限制 | 任意语言都可读写文件 |
| **调试难度** | 状态在内存中，难复现 | 所有输入输出落盘，易复现 |

### 文件交互的设计原则

1. **约定优于配置**：文件路径和格式是唯一的"API"
2. **状态外置**：所有状态存储在文件中，进程无状态
3. **原子写入**：文件操作保证数据一致性
4. **元数据分离**：主体内容 + 元数据，兼顾人类和机器的阅读需求

### 文件结构的演进性

当前的文件结构是为"单任务线性执行"设计的。未来可以平滑扩展：

```
现在：
.agent_workspace/
├── state.json                 # 单任务状态

未来：
.agent_workspace/
├── state.json                 # 全局状态
├── tasks/
│   ├── queue.json             # 任务队列
│   └── history/               # 已完成任务归档
└── sessions/
    └── 2026-04-10/            # 按日期隔离的会话
```

## 角色分工的设计逻辑

### Planner：方向锚定者

**核心职责**：将模糊需求转化为可执行的计划

**设计要点**：

- 必须输出**文件级**的变更清单，不能只说"创建一个函数"
- 必须声明**假设**，让下游 Agent 知道哪些是推断的
- 必须给出**下一步提示**，指导 Coder 的注意力

**为什么重要**：Planner 的输出是整个流程的"锚点"。如果这一步模糊，后续所有环节都会偏离。

### Coder：执行者

**核心职责**：严格按计划生成代码

**设计要点**：

- 必须接收来自三个方向的输入：Planner 的计划（正向）、Debugger 的反馈（反向）和 Reviewer 的评审意见（全局视角）
- 必须记录**已知限制**，诚实声明哪些边界情况未处理
- 必须响应反馈时说明**如何修正的**，形成闭环
- 代码文件解析采用 4 级降级策略（原始格式 → 代码块+注释 → 标题+代码块 → 提取所有代码块），应对 LLM 输出格式的不确定性

**为什么重要**：Coder 是唯一产生代码的角色，其输出的质量直接决定最终产物。

### Debugger：纠偏者

**核心职责**：以批判性视角审查代码

**设计要点**：

- 必须输出**结构化的错误信息**（错误指纹、分类、分析），供系统判断
- 必须给出**可操作的修复建议**，而非只指出问题
- 必须生成**错误指纹**，用于死循环检测

**为什么重要**：Debugger 是质量的第一道防线，也是反馈循环的起点。

### Reviewer：守门人

**核心职责**：对照原始需求做最终裁决

**设计要点**：

- **必须对照原始需求**，这是防止目标漂移的关键
- 评审标准高于 Debugger：不仅看"是否正确"，还看"是否优雅"
- 只有两个输出：批准或驳回（含修改意见）

**为什么重要**：Reviewer 是流程的最后关卡，确保交付物符合最初的承诺。

## 状态机设计

### 状态定义

```
     ┌─────────┐
     │planning │
     └────┬────┘
          ↓
 ┌────→┌─────────┐←──┐
 │     │ coding  │   │
 │     └────┬────┘   │
 │          ↓        │
 │     ┌─────────┐   │ (失败时返回)
 │     │debugging│   │
 │     └────┬────┘   │
 │          ↓        │
 │      ┌──┴──┐     │
 │      ↓     ↓     │
 │     通过   失败───┘
 │      ↓
 │  ┌─────────┐
 │  │reviewing│
 │  └────┬────┘
 │       ↓
 │   ┌──┴──┐
 │   ↓     ↓
 │ 批准   驳回──┘ (驳回时返回 coding)
 │   ↓
 │ ┌─────────┐
 │ │completed│
 │ └─────────┘
```

### 状态转换的触发条件

| 当前状态 | 下一状态 | 触发条件 |
|:---|:---|:---|
| planning | coding | Planner 成功生成计划 |
| coding | debugging | Coder 成功生成代码 |
| debugging | coding | Debugger 发现错误，且未达重试上限 |
| debugging | reviewing | Debugger 验证通过 |
| reviewing | coding | Reviewer 驳回（需要修改） |
| reviewing | completed | Reviewer 批准 |

### 重试计数与循环控制

```json
{
  "retry_count": 0,
  "max_retries": 3,
  "total_cycles": 0,
  "review_reject_count": 0,
  "last_error_signature": "TypeError_line_15"
}
```

- `retry_count` 在每次 Debugger 失败时递增，达到 `max_retries` 后由哨兵检查拦截
- `total_cycles` 在每次进入 Coder 时递增，达到 `TOTAL_CYCLE_LIMIT`（默认 10）后硬性停止，防止 Reviewer 驳回导致的无限循环
- `review_reject_count` 记录 Reviewer 驳回次数，Reviewer 驳回时**不重置** `retry_count`
- `last_error_signature` 用于检测是否在原地打转

## 防御机制详解

### 1. 哨兵检查

**设计动机**：在消耗 Token 调用 LLM 之前，先用廉价的方式检查前置条件。

**检查点清单**：

| 检查点 | 条件 | 失败动作 |
|:---|:---|:---|
| Planner 前 | 任务文件存在且非空 | 终止，提示创建任务 |
| Coder 前 | 计划文件存在且非空 | 回退到 planning |
| Coder 前 | `retry_count` 未达 `max_retries` | 终止，请求人工介入 |
| Coder 前 | `total_cycles` 未达 `TOTAL_CYCLE_LIMIT` | 终止，请求人工介入 |
| Debugger 前 | code 目录有 Python 文件（递归搜索含子目录） | 回退到 coding |
| Reviewer 前 | Debugger 判决为 passed | 回退到 debugging |

**价值**：每次检查可节省一次 LLM 调用（约 0.01-0.1 美元），同时避免 Agent 在无效状态下运行。

### 2. 死循环检测

**设计动机**：AI 可能反复生成相同的错误代码，形成"修复-失败-修复"的死循环。

**检测算法**：

```python
if last_error_signature == current_error_signature:
    consecutive_same_error += 1
else:
    consecutive_same_error = 1

if consecutive_same_error >= LOOP_DETECTION_THRESHOLD:
    trigger_pause()
```

**错误指纹的生成**：系统使用程序化 `generate_error_signature()` 生成确定性 MD5 指纹（取报告前 100 字符的哈希前 16 位），而非依赖 LLM 自述格式。这确保了指纹的稳定性和可比较性。

**阈值选择**：设为 2 次，意味着"连续两次相同错误"就触发警告。这个值基于经验：如果 AI 第一次没修复，第二次大概率也修不好。

### 3. 需求锚点

**设计动机**：经过多轮传递，最初的需求可能被扭曲或遗忘。

**实现方式**：

1. 初始化时，将用户原始需求存入 `state.json` 的 `original_requirement` 字段
2. 该字段只读，任何 Agent 不得修改
3. Reviewer 的 Prompt 中强制包含此字段，要求对照评审

**效果**：无论中间经过多少轮，最终评审时都会回溯到最初的承诺。

### 4. 决策日志

**设计动机**：当出现问题时，需要知道"谁、在什么时候、做了什么决定、为什么"。

**日志条目示例**：

```json
{
  "agent": "Coder",
  "timestamp": "2026-04-10T10:35:00Z",
  "version": 2,
  "changes_summary": "修复了 n=0 时的边界条件错误",
  "addressed_feedback": "在访问缓存前增加了键存在性检查"
}
```

**用途**：

- **调试**：快速定位问题发生在哪个环节
- **审计**：追溯某个决策的来源
- **优化**：分析哪个 Agent 是瓶颈

## 元数据系统的设计

### 为什么需要元数据

Agent 产出的主体内容（如代码、报告）是为人类阅读设计的，但系统需要机器可读的信息来做决策。

例如：

- 人类看测试报告知道"失败了"，系统需要 `"verdict": "failed"` 来判断下一步
- 人类看代码知道"这是修复版"，系统需要 `"addressed_feedback": "..."` 来验证闭环

### 双轨输出模式

每个 Agent 必须同时输出：

```
主体内容（人类可读）        +        元数据（机器可读）
plan_v1.md                          plan_v1.md.meta.json
test_result.md                      test_result.md.meta.json
review.md                           review.md.meta.json
```

注意：元数据文件名 = 主体文件全名 + `.meta.json` 后缀，由 `save_meta()` 自动生成。

### 元数据的强制机制

Prompt 中要求 Agent 用 `[META_START]` 和 `[META_END]` 包裹元数据：

```
[META_START]
{
  "agent": "Planner",
  "key_interpretation": "..."
}
[META_END]
```

程序通过正则解析这部分，失败时有降级策略：

1. 先尝试直接 JSON 解析
2. 若失败，自动修复常见问题（如尾随逗号）后重试
3. 若仍失败，Debugger/Reviewer 从文本关键词推断判定结果（如检测"通过"/"批准"等关键词）

### 元数据 Schema 的设计原则

- **最小化**：只包含系统决策必需字段，避免 Agent 输出冗余信息
- **类型明确**：使用 JSON 保证可解析
- **向后兼容**：新增字段不影响旧版本解析

## 设计权衡与边界

### 有意识的选择

| 选择 | 理由 | 代价 |
|:---|:---|:---|
| 文件交互而非框架 | 透明、可中断、低学习成本 | 无内置并行、无动态协商 |
| 四角色而非动态角色 | 职责清晰、易于理解 | 对某些任务可能过度分工 |
| 纯对话式调试 | 安全、无需沙箱 | 无法发现运行时错误 |
| 单任务线性执行 | 简单可控 | 无法并行处理多任务 |
| 状态全量落盘 | 可随时恢复 | 大量小文件，但对现代文件系统不是问题 |

### 明确的边界（不做什么）

- **不执行真实代码**：Debugger 目前只做静态分析，不运行代码。这是出于安全考虑。未来可作为可选功能添加 Docker 沙箱模式。
- **不自动修改架构**：Planner 不会主动修改项目架构。架构约束需通过 `architecture.md` 人工提供。
- **不做跨任务依赖管理**：当前版本假设任务是独立的。任务间的依赖需要人工管理。
- **不做 Agent 间协商**：当前是单向流水线（带反馈循环），不支持 Agent 之间平等协商。例如 Coder 不能主动要求 Planner 修改计划。

### 何时应该切换到框架模式

当你的需求超出以下边界时，文件交互模式的收益会递减，应考虑迁移到 MS Agent Framework 或 AG2：

- 需要 Agent 间动态协商（如 Coder 和 Planner 讨论方案可行性）
- 需要多任务并行执行
- 需要全局共享记忆（所有 Agent 访问同一个向量数据库）
- 需要复杂的条件分支（根据中间结果动态选择不同的 Agent 路径）

## 总结

本系统的设计核心是**务实**：

- 不强求 AI 解决所有问题，而是设计一个**受控的半自动系统**
- 不追求框架的"高级特性"，而是用**简单透明的文件交互**
- 不假设 AI 完美无误，而是**内置多层防御机制**

这种设计哲学使得系统：

- **易于理解**：任何开发者 10 分钟能看懂流程
- **易于调试**：打开 `.agent_workspace` 目录，一切尽在眼前
- **易于扩展**：新增一个 Skill 就是新增一个 Python 文件
- **成本可控**：哨兵检查防止 Token 浪费

最终，这是一个为**人类与 AI 协作**而设计的系统，而非一个试图取代人类的"黑盒"。
